# The template contract

This file is the agreement between the templates built here and the repacker in
NarraLeaf Studio. Studio rewrites these templates entirely in TypeScript, with
no Android or Apple toolchain on the author's machine — which is only possible
because the templates hold to the shapes below.

**Every rule here is enforced.** The CI (`.github/workflows/build-shells.yml`)
asserts it at build time, and Studio's repacker throws rather than degrading if
a template arrives violating it. Nothing here is stylistic.

The machine-readable half is `manifest.json`, generated by
`scripts/make-manifest.js` from the *built* templates. Studio validates its
`schemaVersion` before anything else, so a mismatch reports "Studio and the
shell template are incompatible" instead of misplacing bytes.

- Current `schemaVersion`: **1**
- Current `shellConfigSchemaVersion`: **1**
- Studio's authority for this type: `src/main/buildWorker/mobile/mobileShellManifest.ts`

## Universal rules

1. **Templates ship unsigned.** Studio signs the Android artifact with a
   generated debug identity; the iOS artifact stays unsigned for the author to
   re-sign.
2. **No symlinks, anywhere.** The repacker's zip writer emits regular files and
   directories only; a symlink would be silently flattened into a file holding
   its target path, so the repacker rejects one instead.
3. **Icon slots are enumerated from the built artifact**, never hand-written.
   The Android build rewrites density directories (`res/mipmap-mdpi/` becomes
   `res/mipmap-mdpi-v4/`), so a source-tree constant would name an entry the
   APK does not contain — and Studio errors on an icon slot it cannot find,
   rather than silently shipping the placeholder icon the author replaced.
4. **Placeholder identity.** Both templates are built with
   `com.narraleaf.shell.placeholder` and visible placeholder names/versions. A
   template that reaches a device unrepacked is meant to look obviously
   unfinished.

## Android

Built by `android/`, output `template.apk` (release) and `template-debug.apk`.

| Rule | Why |
| --- | --- |
| `AndroidManifest.xml` and `resources.arsc` both present | The two patch targets |
| `android:label` is a **literal string**, never `@string/…` | A reference lives in `resources.arsc`, out of the manifest patcher's reach |
| `android:versionCode` is a typed int; `android:versionName` a string | The patcher writes the int in place and the string through the pool |
| `resources.arsc` declares **exactly one** package | The patcher refuses to guess which to rename |
| **Zero `<provider>`** in the merged manifest | An authority embeds the application id as a prefix; two games repacked from one template would collide with `INSTALL_FAILED_CONFLICTING_PROVIDER` |
| **Zero third-party dependencies** | Any library risks dragging in a `<provider>` (androidx startup/profileinstaller do), and grows a template that ships inside every game |
| Icons are plain mipmap PNGs, not adaptive-icon XML | The repacker swaps PNG bytes; it cannot compile new resources |
| `minifyEnabled = false` | Shrinking moves the entry names and manifest shapes the CI asserted |
| No `android:screenOrientation` | Orientation is applied at runtime from `shell-config.json`, so no manifest attribute needs patching |

**Injection points** (`manifest.json` → `android`):

- `wwwRoot: "assets/www/"` — the payload, as **zip entry paths**.
- `shellConfigPath: "assets/shell-config.json"`.

> Note the asset path duality: the repacker writes the zip entry
> `assets/www/index.html`, while the shell opens it through `AssetManager` as
> `www/index.html` (the `assets/` prefix is the zip's, not the manager's). The
> manifest always states the **zip entry path**.

The placeholder application id is renamed pool-wide by Studio under an exact +
dotted-prefix rule, so any string derived from it follows. The zero-`<provider>`
rule stands regardless: it removes the failure mode rather than patching around
it.

Two rules that are **not** the template's to keep, listed because they are easy
to mistake for its business: `resources.arsc` must ship **stored and 4-byte
aligned** in the finished APK (Android 11+ rejects the install otherwise), and
every other stored entry must be aligned too. Both are the repacker's
guarantees — it reads the template's table however it was encoded and always
writes it back stored and aligned — so the template may store or deflate as its
build tools please.

## iOS

Built by `ios/`, output `template.app.zip` (release) and `template-debug.app.zip`.

| Rule | Why |
| --- | --- |
| Archived with `ditto -c -k --keepParent`, so every entry is prefixed with `Shell.app/` | The repacker re-lays this prefix into `Payload/<GameName>.app/` |
| `Info.plist` is **plain text XML**, at the `.app` root | The repacker rewrites text plists; it cannot patch a binary one. `xcodebuild` converts plists to binary by default — the CI asserts this one did not |
| `Info.plist` has all of: `CFBundleIdentifier`, **`CFBundleDisplayName`**, `CFBundleShortVersionString`, `CFBundleVersion`, `UISupportedInterfaceOrientations` | Each is patched by key; a missing key is an error, not a skip. Without `CFBundleDisplayName` the game shows the shell's name on the home screen |
| Executable named `Shell`, mode 0755 | The repacker preserves the bit verbatim; without it the app cannot launch |
| **No embedded frameworks** | They bring symlinks, which rule 2 forbids |
| Icons are plain PNGs named from `CFBundleIcons`, not an asset catalog | `Assets.car` can only be produced by `actool`; an offline repack cannot |

**Injection points** (`manifest.json` → `ios`), all relative to the `.app`:

- `wwwRoot: "www/"`
- `shellConfigPath: "shell-config.json"`

## `shell-config.json`

Written by the repacker into each template at `shellConfigPath`; read by the
shell at launch. Every field has a default, so a version skew degrades rather
than crashes.

```jsonc
{
    "schemaVersion": 1,
    "orientation": "landscape", // landscape | portrait | auto
    "backgroundColor": "#000000", // the pre-boot background, matching the web export
    "contentKey": "…" // optional; present only when the payload is encoded
}
```

`contentKey` is an opaque token the shell hands to the decoder library. When it
is present the whole payload is encoded and the shell serves nothing without
decoding it; when it is absent the payload is plain and served verbatim. It is
all-or-nothing: a shell asked for the length of a file cannot see that file's
bytes, so it cannot tell protected files apart one by one.

## The decoder libraries

The shells carry no decoding logic. `scripts/pull-decoder.js` fetches prebuilt
libraries from the published `@narraleaf/encryption` package into
`android/app/src/main/jniLibs/` and `ios/Vendor/`; both are build inputs and
neither is committed. CI pulls them before building, and asserts they reached
the template — a shell that builds without them fails only on a device, the
first time it meets an encoded payload.

- **Android**: one `.so` per ABI (`arm64-v8a`, `armeabi-v7a`, `x86_64`),
  exporting the JNI entry points `NativeContentDecoder` declares. **The class
  and method names there are an ABI contract with the library**; renaming them
  breaks the lookup at runtime, on a device, not here.
- **Android packaging**: `extractNativeLibs="true"` and deflated `.so` entries,
  both deliberate. Uncompressed libraries are mmap'd straight from the APK and
  must be page-aligned (16 KB on Android 15+), but the Studio repacker rewrites
  the archive 4-byte aligned — a page-aligned layout would not survive it, and
  the failure would surface as an install or load error on a device. Extracting
  costs a few tens of KB at install and keeps the repack contract whole.
- **iOS**: a static `NlCrypto.xcframework`. Static, not dynamic: a dynamic
  framework inside an `.app` brings embedded frameworks and symlinks, and the
  repack rejects symlinks outright.

## Serving contract

Both shells must serve the payload to their web view with:

- **HTTP Range** (`206`, exact `Content-Range`, suffix `bytes=-N` and
  open-ended `bytes=N-` forms, `416` when unsatisfiable, `Accept-Ranges: bytes`).
  Without it audio and video cannot seek — this is the single most important
  rule here.
- **Streaming reads.** A game's video is routinely larger than the heap.
- **Correct `Content-Type`**, from the shared table in `Mime.kt` / `Mime.swift`.
  A module script served as the wrong type never executes.
- **A secure context**, so IndexedDB — where the game's saves live — works.
  Android intercepts an `https://` origin under the reserved `.localhost` TLD;
  iOS must use a custom scheme, because WebKit refuses to hand a
  `WKURLSchemeHandler` a scheme it implements itself.
- **Media unlocked**: Android `mediaPlaybackRequiresUserGesture = false`; iOS
  `mediaTypesRequiringUserActionForPlayback = []` plus
  `allowsInlineMediaPlayback = true`. Otherwise the opening track never sounds
  and every video is yanked into the system's fullscreen player.
- **Bytes pass through `ContentDecoder`** on their way out. The default is the
  identity implementation. The interface is bytes in, bytes out and nothing
  more; it is told nothing about the request.

`test-payload/` asserts all of the above on a device. A shell that fails it is
not done, whatever else builds.

## Debug variants

Both platforms publish a debug template alongside the release one. Studio uses
the debug variant **only when running unpackaged** (a developer build); a
packaged Studio always repacks the release template. Release templates never
enable web-view inspection.
